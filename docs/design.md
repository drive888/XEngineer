# AI 语音绘图工具设计文档

本文说明 AI 语音绘图工具的目标、指令能力、实现状态和未完成原因。文档面向评审和后续开发，重点回答三个问题：计划支持什么，当前已经做到什么，后续还差什么。

## 1. 项目目标

本项目要做一款以语音为主要入口的绘图工具。你点击语音按钮或按空格开始录音，说出绘图需求，再次点击按钮或按空格结束录音，系统把语音转成文本、解析成绘图操作，并在画布上逐步执行。

核心目标：

- **语音优先**：创建、清空、撤销、修改等主要操作都能通过语音完成
- **可理解**：自然语言先转成结构化绘图操作，再交给画布执行
- **有过程**：绘图不是一次性贴图，而是以笔画、路径和动画逐步呈现
- **可恢复**：误识别或误操作后，可以清空、撤销、重做
- **可扩展**：保留本地规则、AI 解析、资产绘制和画笔绘制多条路线

## 2. 使用流程

用户流程保持短路径：

1. 进入产品介绍页，点击进入工作区
2. 点击底部语音按钮，或按一次空格开始录音
3. 说出指令，例如“画一棵树在草地上”
4. 再次点击按钮，或再按一次空格结束录音
5. 系统识别语音，解析指令，并在画布上绘制
6. 继续用语音修改，例如“清空画布后画一个火箭”

## 3. 系统架构

当前架构把语音、理解和绘图拆开，便于替换模型或画布实现。

```text
语音输入
  -> MiMo ASR / Web Speech 兜底
  -> 本地规则解析 / MiMo AI 解析
  -> DrawOperation[]
  -> 布局与引用解析
  -> tldraw 画布执行
  -> Rough SVG 动画层展示绘制过程
```

主要模块：

| 模块 | 当前实现 | 作用 |
|---|---|---|
| 语音识别 | MiMo `mimo-v2.5-asr`，Web Speech 兜底 | 把语音转成文本 |
| 指令解析 | 本地规则 + MiMo `mimo-v2.5-pro` | 把文本转成绘图操作 |
| 画布底座 | React + tldraw | 管理对象、选择、撤销、重做 |
| 绘制效果 | Rough SVG + 路径动画 | 让图形有手绘感和绘制过程 |
| 服务端 | Node.js + Express | 代理 ASR、AI parser 和模型调用 |

## 4. 计划支持的指令能力

计划支持的能力按优先级分成五类。

| 类型 | 示例 | 目标 |
|---|---|---|
| 基础图形 | “画一个红色圆形” | 创建圆形、矩形、箭头、文本等基础对象 |
| 常见物体 | “画一棵树”“画一个火箭” | 绘制树、汽车、云、太阳、火箭等对象 |
| 场景表达 | “画一片草原”“画一棵树在草原上” | 组合背景和物体，减少重叠 |
| 图示表达 | “画一个关于 AI 的思维导图” | 创建中心节点、分支、文字和连接线 |
| 画布管理 | “清空画布”“撤销”“重做” | 支持恢复和重新创作 |

复杂指令需要拆解成多步操作。例如“先画一棵树，清除画布后，画一个火箭”应拆成：

1. 创建树
2. 清空画布
3. 创建火箭

执行时需要按顺序播放，不能跳到最后一帧。

## 6. 当前限制

这些问题还没有完全解决，需要后续迭代。

| 问题 | 当前表现 | 原因 |
|---|---|---|
| 不是真实时 | 需要录完一句再执行 | MiMo 当前接入方式是整段音频请求 |
| Web Speech 不稳定 | 浏览器兼容性和识别准确率受限 | 浏览器原生能力差异大 |
| 开放物体质量不稳定 | 生僻物体可能画得抽象 | 规则和画笔模板覆盖不足 |
| 复杂布局仍可能重叠 | 多对象空间关系偶尔不准 | 布局模型还没有独立优化 |
| AI 画笔控制还在试验 | 不能稳定生成高质量笔画 | 大模型输出路径需要约束和校验 |
| Realtime 模型未接入 | Gemini Live 被拒绝访问，OpenAI Realtime 配额不足 | 账号权限和成本限制 |

## 7. 容错与延迟策略

系统采用“本地优先，AI 补充”的策略。

- **低延迟指令**：清空、撤销、基础形状走本地规则，减少等待
- **开放表达**：复杂描述交给 MiMo AI parser，得到结构化操作
- **失败兜底**：MiMo ASR 失败时尝试 Web Speech
- **可撤销**：误画后通过“撤销”或“清空画布”恢复
- **可观察**：界面显示识别文本和执行状态，方便定位问题

后续要增强两点：

- 加入置信度判断，低置信度时先确认再画
- 加入布局规划器，让“旁边”“上方”“不重叠”等空间关系更稳定

## 8. 绘图表现方案

当前不采用纯图片生成。目标是让系统像在画，而不是插入图片。

现有路线：

- 使用矢量对象承载结果，方便编辑和撤销
- 使用 Rough SVG 生成手绘线条质感
- 使用路径动画模拟一笔一画出现
- 对常见物体使用可解释的部件组合，例如树干、树冠、草线、火箭尾焰

后续路线：

- 让 AI 输出受约束的画笔路径，而不是直接输出图片
- 对路径做合法性检查，避免乱线、重叠和画布越界
- 建立更多物体的部件库，提升开放命令质量

## 10. 后续计划

下一阶段优先做三件事：

1. **布局规划器**：让多个对象自动避让，稳定处理“旁边”“上方”“左边”
2. **AI 画笔路径**：让模型输出绘制步骤，再由画布按步骤画出来
3. **实时语音链路**：等账号权限和成本条件满足后，接入可用的实时语音模型
