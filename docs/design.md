# AI 语音绘图工具设计文档

## 1. 项目概述

本项目面向“题目二：AI 语音绘图工具”，目标是开发一款纯语音控制的绘图应用。用户不能使用鼠标或键盘进行创作，只能通过语音指令完成图形创建、属性修改、对象选择、布局调整、删除、撤销、重做和画布管理。

项目重点不是复刻专业设计软件，而是验证一条完整、可解释、可恢复的智能绘图链路：

```text
语音输入
  -> 语音识别
  -> 文本标准化
  -> 指令理解
  -> 复杂指令拆解
  -> 结构化绘图操作
  -> 画布执行
  -> 语音和界面反馈
```

核心目标：

- 纯语音优先：主要创作流程必须能在不使用鼠标和键盘的情况下完成。
- 低延迟优先：简单指令使用本地规则快速执行，复杂指令再调用 AI 解析。
- 可解释执行：所有自然语言指令都先转成结构化操作，再由画布执行器执行。
- 容错可恢复：识别错误、理解歧义和误操作都可以通过确认、撤销、重做和日志恢复。
- 文档可追踪：明确记录计划支持的指令、最终实现的能力和未完成原因。

## 3. 技术选型

| 模块 | 选型 | 原因 |
|---|---|---|
| 前端框架 | React + Vite + TypeScript | 开发速度快，类型约束清晰，适合课程项目交付 |
| 画布底座 | `tldraw` | 提供成熟无限画布、图形对象、文本、箭头、历史记录和 Editor API |
| 语音识别 | Web Speech API + OpenAI Transcribe 兜底 | 浏览器语音低延迟，云端转写提高演示稳定性 |
| 指令解析 | 本地规则 + AI JSON 解析 | 简单指令快，复杂指令智能 |
| 语音反馈 | SpeechSynthesis | 浏览器原生能力，成本低 |
| 服务端 | Node.js + Express | 用于 OpenAI 转写和 AI 解析接口 |
| 数据协议 | 自定义 DrawOperation JSON | 让指令理解和画布执行解耦 |

### 3.1 为什么使用 tldraw

本项目不继续手写 SVG 画布，原因是题目评分重点在语音理解、容错、复杂指令拆解和执行链路，而不是底层画布编辑器。

tldraw 适合作为画布底座：

- 图形、文本、箭头等基础对象已内置。
- 支持缩放、平移、选择、撤销、重做等常见画布能力。
- 提供 Editor API，适合通过程序创建、修改、删除对象。
- 后续可以扩展自定义 shape 和语义 metadata。

需要注意：

- 不直接依赖 `tldraw/agent-template` 作为项目底座，因为它更像官方 AI 示例，社区验证较少。
- 本项目使用成熟的 `tldraw` 主 SDK，只参考 AI agent 示例中的结构化操作思路。
- 为满足“纯语音控制”，需要隐藏或弱化鼠标工具入口，主要通过语音触发画布操作。

## 5. 指令能力设计

### 5.1 图形创建

计划支持的基础图形：

- 圆形：“画一个红色圆形”
- 矩形：“画一个蓝色矩形”
- 圆角矩形：“画一个圆角按钮”
- 三角形：“画一个黄色三角形”
- 菱形：“画一个判断节点”
- 直线：“画一条横线”
- 箭头：“从左到右画一条箭头”
- 文本：“写上欢迎使用”

计划支持的组合图形：

- 按钮：“画一个登录按钮”
- 标签：“画一个红色标签”
- 流程图：“画一个开始到结束的流程图”
- 简单场景：“画一个太阳和一棵树”

### 5.2 属性控制

计划支持：

- 颜色：红、蓝、绿、黄、黑、白、灰、紫、橙、粉。
- 大小：小、中等、大、更大、更小。
- 位置：左上、上方、右上、左侧、中间、右侧、左下、下方、右下。
- 样式：填充色、边框色、边框粗细。
- 文本：内容、字号、文本颜色。

示例：

- “在右下角画一个小的绿色圆形。”
- “把它的边框改成黑色。”
- “把文字调大一点。”
- “把这个图形改成透明填充。”

### 5.3 编辑和变换

计划支持：

- 移动：“向右移动一点”“移动到左上角”
- 缩放：“放大一点”“缩小一点”
- 改色：“把它改成绿色”
- 改文字：“把文字改成提交”
- 删除：“删除刚才那个”
- 复制：“复制当前对象”
- 层级：“置顶”“置底”

### 5.4 对象引用

计划支持的引用方式：

| 引用方式 | 示例 |
|---|---|
| 最近对象 | “刚才那个”“它”“这个” |
| 当前选中 | “当前对象”“选中的图形” |
| 类型引用 | “圆形”“矩形”“文字” |
| 属性引用 | “红色圆形”“蓝色矩形” |
| 位置引用 | “左边的圆”“右上角的文字” |
| 尺寸引用 | “最大的矩形”“最小的圆” |
| 编号引用 | “第一个图形”“编号二” |

引用解析优先级：

1. 明确编号或 id。
2. 当前选中对象。
3. 最近创建或最近编辑对象。
4. 类型、颜色、位置、尺寸等条件匹配。
5. 多个候选时请求用户确认。

### 5.5 画布管理

计划支持：

- 撤销：“撤销”“退回上一步”
- 重做：“重做”“恢复上一步”
- 清空：“清空画布”
- 导出：“导出图片”“保存作品”
- 重置视图：“回到中心”“重置视图”

## 6. 指令解析方案

### 6.1 文本标准化

语音识别结果先经过标准化处理：

| 原始表达 | 标准化 |
|---|---|
| 圆圈 | 圆形 |
| 方块、长方形 | 矩形 |
| 兰色 | 蓝色 |
| 删掉、去掉 | 删除 |
| 退回上一步 | 撤销 |
| 恢复上一步 | 重做 |

标准化只负责清洗和归一，不直接执行操作。

### 6.2 本地规则解析

本地规则覆盖高频和确定性强的命令：

- 创建基础图形。
- 解析颜色、大小、位置。
- 修改最近对象颜色。
- 上下左右移动。
- 放大、缩小。
- 删除、撤销、重做、清空。
- 简单多步拆句。

本地规则优点：

- 延迟低。
- 稳定可控。
- 易于解释。
- 不依赖网络和 API。

### 6.3 AI 语义解析

AI 用于处理本地规则难以覆盖的指令：

- 复杂组合图形。
- 长句多步操作。
- 模糊的自然语言意图。
- 场景化绘图请求。

约束：

- AI 只能返回 JSON。
- AI 不能直接操作画布。
- AI 输出必须经过 schema 校验。
- AI 生成的操作必须属于允许的操作集合。
- 低置信度时返回确认问题。

示例输出：

```json
{
  "confidence": 0.86,
  "needsConfirmation": false,
  "question": "",
  "operations": [
    {
      "action": "create",
      "kind": "shape",
      "shape": "rectangle",
      "fill": "blue",
      "position": "center",
      "size": "large"
    }
  ]
}
```

### 6.4 指令路由策略

| 指令类型 | 示例 | 路由 |
|---|---|---|
| 简单创建 | “画一个红色圆形” | 本地规则 |
| 简单编辑 | “把它改成蓝色” | 本地规则 |
| 历史操作 | “撤销” | 本地规则 |
| 简单多步 | “画一个圆，然后向右移动一点” | 本地拆句 |
| 模板图形 | “画一个登录按钮” | 模板优先，AI 兜底 |
| 复杂语义 | “画一个用户注册流程图” | AI 解析 |
| 歧义对象 | “把那个变大” | 引用解析或确认 |

## 7. 结构化操作协议

所有解析器最终输出统一的 `DrawOperation[]`。

```ts
type DrawOperation =
  | {
      action: 'create'
      kind: 'shape' | 'text' | 'arrow'
      shape?: 'rectangle' | 'ellipse' | 'triangle' | 'diamond'
      text?: string
      fill?: string
      stroke?: string
      position?: PositionName
      size?: SizeName
      target?: TargetRef
    }
  | {
      action: 'update'
      target: TargetRef
      fill?: string
      stroke?: string
      text?: string
    }
  | {
      action: 'move'
      target: TargetRef
      direction?: 'left' | 'right' | 'up' | 'down'
      dx?: number
      dy?: number
    }
  | {
      action: 'resize'
      target: TargetRef
      scale: number
    }
  | {
      action: 'delete' | 'select'
      target: TargetRef
    }
  | {
      action: 'undo' | 'redo' | 'clear' | 'export'
    }
```

目标引用：

```ts
type TargetRef =
  | { type: 'last' }
  | { type: 'selected' }
  | { type: 'byId'; id: string }
  | {
      type: 'query'
      shape?: string
      color?: string
      position?: string
      order?: 'largest' | 'smallest' | 'first' | 'last'
    }
```

设计原因：

- 方便日志记录。
- 方便测试解析结果。
- 方便本地规则和 AI 解析复用同一个执行器。
- 避免 AI 直接修改画布导致不可控行为。

## 8. 画布执行方案

画布执行器负责把 `DrawOperation` 转换为 tldraw Editor API 调用。

主要职责：

- 根据位置词计算画布坐标。
- 根据大小词计算图形尺寸。
- 创建 tldraw shape。
- 更新 shape 属性。
- 删除 shape。
- 调用撤销、重做、清空。
- 维护最近对象和语义标签。

上下文状态：

```ts
type VoiceDrawingContext = {
  lastShapeId: string | null
  selectedShapeIds: string[]
  recentShapeIds: string[]
  pendingConfirmation: PendingConfirmation | null
  semanticIndex: Record<string, ShapeMetadata>
}
```

`semanticIndex` 用于记录图形的语义信息，例如“红色圆形”“登录按钮”“开始节点”，方便后续语音选择。

## 9. 延迟优化方案

为了满足语音到绘图操作的响应延迟要求，采用以下策略：

- 简单指令优先走本地规则，避免等待 AI。
- 语音识别得到 final result 后立即进入解析。
- 本地规则解析同步完成，直接更新画布。
- 复杂指令解析期间显示“正在理解”状态。
- AI 请求设置超时，失败后给出可恢复反馈。
- 常用模板本地化，例如按钮、流程图、太阳等。
- 操作执行分步完成，复杂指令每执行一步都记录日志。

目标：

- P0 简单指令在 1 秒内完成画布反馈。
- 复杂 AI 指令允许稍慢，但必须显示处理中状态。
- AI 不可用时，基础绘图仍然可用。

## 10. 容错方案

### 10.1 识别容错

- 支持同义词归一。
- 支持常见错别字修正。
- 显示原始识别文本，便于用户发现错误。
- 识别失败时提示用户重说。

### 10.2 理解容错

- 本地规则返回置信度。
- AI 返回置信度和确认问题。
- 无法理解时不执行画布操作。
- 多候选对象时进入确认流程。

示例：

```text
用户：把红色圆形放大
系统：找到两个红色圆形，你想放大左边的还是右边的？
用户：左边的
系统：已放大左边的红色圆形
```

### 10.3 执行容错

- 创建、修改、删除都进入历史记录。
- 支持“撤销”和“重做”。
- 删除前记录目标对象。
- 执行失败时保留原画布状态。

### 10.4 反馈容错

- 界面状态展示当前阶段：正在听、正在理解、已执行、需要确认、执行失败。
- 关键反馈用语音播报。
- 操作日志记录原始语音、标准化文本、结构化操作和执行结果。

### 12.1 总体工作流 Skill

| 使用时机 | 使用的 skill | 用途 |
|---|---|---|
| 需要继续补充专项能力时 | `find-skills` | 根据项目缺口搜索可用 skill，并按安装量、来源和仓库质量筛选 |
| 确认要新增 skill 时 | `skill-installer` | 将选中的 skill 安装到本地 Codex 环境，安装后重启 Codex 以加载 |

## 16. 当前结论

项目当前采用“tldraw 主画布 + SVG 回归镜像 + 自研语音指令链路”的方案：

- 用 tldraw 渲染主画布，用户看到的是 tldraw canvas；用自维护状态和 SVG 镜像保证本轮绘图闭环、测试稳定性、导出和可解释执行。
- 用本地规则保证 P0 指令低延迟和高稳定性。
- 用 MiMo ASR/OpenAI 转写增强非 Chrome 浏览器语音入口，不把基础绘图执行依赖在浏览器 Web Speech 上。
- 用 AI parser 增强复杂自然语言意图，不把基础功能依赖在 AI 上；AI 输出必须先转成受控 `DrawOperation[]`，再由执行器落图。
- 用视觉资产库承接“画某个对象”的需求：AI 或本地解析只负责识别对象意图，渲染层用可复用 SVG stroke 组件保证低延迟和可识别度。
- 所有自然语言都转成结构化操作后再执行。
- 使用日志、确认、撤销和语音播报提升容错性。

这一路线能更好地贴合题目二的核心要求：纯语音控制、准确理解、低延迟响应、复杂指令拆解，以及可解释的最终设计文档。
